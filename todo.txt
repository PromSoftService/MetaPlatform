METAPlatform v1 — ПОЛНОЕ ТЕХНИЧЕСКОЕ ЗАДАНИЕ

Архитектура платформы, проектная модель, модульная система, система команд,
формат документов и структура исходного кода

======================================================================
1. НАЗНАЧЕНИЕ ДОКУМЕНТА
======================================================================

Настоящее техническое задание описывает целевую архитектуру MetaPlatform v1 —
модульной инженерной платформы для разработки, генерации, тестирования
и дальнейшего расширения инструментов АСУ ТП.

Документ фиксирует:
- назначение и границы платформы;
- архитектурную модель системы;
- сущность проекта;
- модульную архитектуру;
- систему команд;
- структуру документов модулей;
- структуру каталогов проекта пользователя;
- структуру каталогов исходного кода самой платформы;
- правила взаимодействия между платформой и модулями;
- состав функциональности для реализации;
- требования к MetaGen как первому рабочему модулю;
- требования к интеграции внешних Python-движков;
- требования к UI, логированию, файловой системе и жизненному циклу документов.

Документ предназначен как основное архитектурное ТЗ для реализации
MetaPlatform без промежуточных упрощённых шагов.

======================================================================
2. КОНТЕКСТ И ОБЩАЯ ИДЕЯ СИСТЕМЫ
======================================================================

Изначально задача заключалась в создании UI для существующего решения
генерации PLC-кода.

В процессе развития стало очевидно, что система фактически перерастает
в полноценную инженерную платформу, которая должна объединять несколько
независимых подсистем.

В рамках MetaPlatform предполагается развитие следующих модулей:

- MetaGen — модуль генерации PLC-кода по инженерным шаблонам;
- MetaLab — модуль виртуальной наладки, simulation / scenario engine;
- MetaView — модуль визуализации, экранов, лёгкой SCADA-логики.

Таким образом, MetaPlatform не является ни только генератором кода,
ни только UI, ни только оболочкой под один сценарий, а представляет
собой контейнер инженерных модулей, объединённых общей средой разработки.

======================================================================
3. ТЕРМИНЫ И ОПРЕДЕЛЕНИЯ
======================================================================

3.1. MetaPlatform
MetaPlatform — общая платформа, контейнер, Workbench,
в который подключаются инженерные модули.

3.2. Модуль
Модуль — функциональная подсистема, расширяющая платформу
в своей предметной области.
Примеры: MetaGen, MetaLab, MetaView.

3.3. Проект
Проект — это директория с файлом описания проекта и набором документов модулей.
Проект является основной единицей хранения пользовательских данных.

3.4. Документ модуля
Документ модуля — файл, принадлежащий конкретному модулю,
редактируемый через платформу.
Например, metagen/pumps.yaml — документ MetaGen.

3.5. Редактор
Редактор — UI-представление документа внутри вкладки Workbench.

3.6. Workbench
Workbench — основная рабочая оболочка MetaPlatform,
включающая панели, вкладки, дерево проекта, редакторы и логи.

3.7. Команда
Команда — единая форма выполнения действия в системе.
Например: открыть проект, создать документ, переименовать документ,
сгенерировать код.

3.8. Внешний движок
Внешний движок — внешняя исполняемая логика, вызываемая из платформы,
но не реализованная внутри её UI-кода.
Для MetaGen таким движком является Python-генератор.

======================================================================
4. ЦЕЛИ СИСТЕМЫ
======================================================================

4.1. Основная цель

Создать модульную инженерную платформу, позволяющую:
- работать с проектами АСУ ТП;
- подключать независимые инженерные модули;
- хранить документы модулей в проекте;
- открывать и редактировать документы в едином Workbench;
- запускать внешние генераторы и обработчики;
- масштабировать систему без архитектурного рефакторинга.

4.2. Практическая цель версии v1

В версии MetaPlatform v1 должна быть реализована полноценная платформа
с первым рабочим модулем MetaGen, при этом архитектура сразу должна быть
подготовлена для подключения MetaLab и MetaView как самостоятельных модулей.

4.3. Архитектурная цель

Система должна строиться не как "один большой UI под один генератор",
а как платформа с независимыми модулями, единым проектом,
единым Workbench и единой системой команд.

======================================================================
5. ОБЩАЯ АРХИТЕКТУРНАЯ МОДЕЛЬ
======================================================================

5.1. Архитектурный принцип

MetaPlatform должна строиться по следующей модели:

MetaPlatform
|
|-- Core Services
|   |-- Logger
|   |-- CommandBus
|   |-- ModuleRegistry
|   |-- ProjectManager
|   |-- FileSystemBridge
|   `-- DocumentLoader
|
|-- Workbench UI
|   |-- Top Panel
|   |-- Project Tree
|   |-- Tabs
|   |-- Editor Host
|   `-- Logs
|
|-- Modules
|   |-- MetaGen
|   |-- MetaLab
|   `-- MetaView
|
`-- External Engines
    |-- Python generator (MetaGen)
    `-- Python simulation engine (MetaLab)

5.2. Главный принцип разделения ответственности

MetaPlatform отвечает за:
- оболочку;
- проект;
- загрузку модулей;
- систему команд;
- открытие документов;
- вкладки;
- дерево проекта;
- общие сервисы;
- логирование;
- файловую систему;
- вызов модульных обработчиков.

Модули отвечают за:
- свои типы документов;
- создание своих документов;
- свои редакторы;
- свои команды;
- свою предметную логику;
- интеграцию со своими внешними движками.

======================================================================
6. ПРИНЦИПЫ ПРОЕКТИРОВАНИЯ
======================================================================

6.1. Платформа не должна зависеть от одного модуля

MetaPlatform не должна быть "MetaGen с приделанными потом MetaLab и MetaView".
Она должна изначально быть независимым контейнером для модулей.

6.2. Проект — основной уровень хранения

В системе первичным контейнером пользовательских данных является проект,
а не вкладка и не runtime текущего редактора.

6.3. Документ модуля — основная единица редактирования

Платформа работает не с абстрактными UI-состояниями,
а с документами модулей, расположенными внутри проекта.

6.4. UI не должен выполнять предметную логику напрямую

UI должен только инициировать команды и отображать результат.
Логика изменения системы должна выполняться обработчиками команд
и сервисами платформы/модулей.

6.5. Все действия проходят через систему команд

Открытие проекта, создание документа, переименование,
генерация кода и прочие действия должны выполняться через Command Bus.

6.6. Внешние движки должны оставаться отделёнными от UI

Python-скрипты и прочие внешние механизмы не должны смешиваться
с кодом UI, а должны вызываться через отдельные мосты.

6.7. Файловая модель должна быть простой

Проект — это папка.
Документы — это YAML-файлы.
Результаты генерации — отдельные файлы в отдельной папке.

6.8. Архитектура должна быть масштабируемой без слома базовой модели

После запуска v1 должно быть возможно:
- добавлять новые типы документов;
- добавлять новые команды;
- добавлять новые модули;
- подключать дополнительные движки;
- развивать MetaLab и MetaView без перестройки MetaPlatform.

======================================================================
7. СУЩНОСТЬ ПРОЕКТА
======================================================================

7.1. Общая идея

Проект в MetaPlatform — это директория, содержащая:
- файл описания проекта project.yaml;
- подпапки документов модулей;
- папку с результатами генерации;
- в будущем — дополнительные служебные данные проекта.

7.2. Структура проекта пользователя

project-root/
|
|-- project.yaml
|
|-- metagen/
|   |-- pumps.yaml
|   |-- valves.yaml
|   `-- conveyors.yaml
|
|-- metalab/
|   |-- startup_scenario.yaml
|   `-- emergency_stop.yaml
|
|-- metaview/
|   |-- main_screen.yaml
|   |-- trends.yaml
|   `-- alarms.yaml
|
`-- generated/
    |-- pumps.st
    |-- valves.st
    `-- conveyors.st

7.3. Обязательные элементы проекта

Обязателен:
- project.yaml

Допустимы пустые каталоги:
- metagen/
- metalab/
- metaview/
- generated/

7.4. Почему проект именно в виде директории

Такая модель выбрана как наиболее подходящая для инженерной среды,
потому что:
- её легко переносить;
- её легко хранить в Git;
- её легко бэкапить;
- её легко анализировать вручную;
- она не требует СУБД;
- она хорошо сочетается с файловыми документами модулей;
- она соответствует инженерному ожиданию "проект = папка".

======================================================================
8. ФОРМАТ ФАЙЛА ПРОЕКТА PROJECT.YAML
======================================================================

8.1. Назначение

project.yaml является корневым описанием проекта и должен содержать:
- идентичность проекта;
- отображаемое имя;
- описание;
- список подключённых модулей;
- служебные метаданные версии формата;
- при необходимости — настройки проекта платформенного уровня.

8.2. Пример

kind: metaplatform.project
version: 1

project:
  id: feedmill_plant
  name: Комбикормовый завод
  description: Проект АСУ ТП комбикормового завода
  createdAt: 2026-03-10
  updatedAt: 2026-03-10

modules:
  - MetaGen
  - MetaLab
  - MetaView

paths:
  metagen: metagen
  metalab: metalab
  metaview: metaview
  generated: generated

meta:
  author: Alex
  company: ""
  tags:
    - feedmill
    - asutp
    - metaplatform

8.3. Обязательные поля

На корневом уровне:
- kind
- version
- project
- modules

Внутри project:
- id
- name

8.4. Роль файла project.yaml

Файл project.yaml не должен хранить полный список документов проекта.
Список документов определяется сканированием папок модулей.

Это решение принимается сознательно, чтобы:
- избежать дублирования данных;
- упростить merge в Git;
- позволить внешнее добавление файлов;
- не синхронизировать руками список файлов и файловую систему.

======================================================================
9. ПРОЕКТНАЯ МОДЕЛЬ ВНУТРИ ПЛАТФОРМЫ
======================================================================

9.1. Project Runtime

После открытия проекта MetaPlatform должна создать внутреннюю runtime-модель проекта,
включающую:
- путь к проекту;
- содержимое project.yaml;
- зарегистрированные активные модули;
- список обнаруженных документов по модулям;
- дерево проекта для UI;
- кэш открытых документов по необходимости.

9.2. Project Manager

Для управления проектом должен существовать сервис:
projectManager.js

Он отвечает за:
- создание нового проекта;
- открытие существующего проекта;
- закрытие проекта;
- сохранение проекта;
- сканирование каталогов модулей;
- предоставление списка документов;
- выдачу информации для Project Tree;
- операции создания, переименования и удаления документов
  совместно с модулями и командами;
- выдачу текущего активного проекта.

9.3. Основные операции Project Manager

ProjectManager обязан поддерживать:
- createProject(...)
- openProject(...)
- saveProject(...)
- closeProject()
- getCurrentProject()
- scanProjectDocuments()
- getDocumentsByModule(moduleId)
- getDocumentByPath(documentPath)
- createDocument(...)
- renameDocument(...)
- deleteDocument(...)

======================================================================
10. ДОКУМЕНТЫ МОДУЛЕЙ
======================================================================

10.1. Общая модель

Каждый модуль хранит свои документы в собственной папке проекта.

Пример:
- документы MetaGen лежат в metagen/
- документы MetaLab лежат в metalab/
- документы MetaView лежат в metaview/

10.2. Принцип

Документ принадлежит модулю не по папке как таковой, а по:
- типу документа (kind);
- расширению или пути хранения;
- обработке зарегистрированным модулем.

10.3. Идентификация

Документ должен иметь:
- путь на файловой системе;
- модуль-владелец;
- kind;
- версию формата;
- имя или идентификатор документа.

======================================================================
11. ФОРМАТ ДОКУМЕНТА METAGEN
======================================================================

11.1. Общий принцип

Для MetaGen принимается решение хранить в одном YAML-документе всё необходимое:
- параметры;
- таблицу данных;
- список инстансов;
- шаблон кода;
- настройки генерации;
- метаданные документа.

Это сознательное архитектурное решение, соответствующее существующей практике
работы с Excel-шаблонами и Python-генератором.

11.2. Назначение MetaGen-документа

MetaGen-документ является полным инженерным шаблоном компонента
или сущности генерации.

Документ должен быть самодостаточным:
- его можно открыть в MetaPlatform;
- его можно редактировать;
- его можно передать во внешний Python-генератор;
- из него можно получить ST-код.

11.3. Структура MetaGen-документа v1

kind: metagen.component
version: 1

component:
  id: pumps
  name: Pumps
  type: template
  module: MetaGen
  description: Каскад насосов

params:
  format: header-plus-rows
  header: []
  rows: []

data:
  format: table
  columns: []
  rows: []

instances:
  format: list
  rows: []

code:
  format: template-text
  language: st-template
  text: ""

generation:
  engine: python
  entrypoint: gen_v2.py
  mode: external
  output:
    language: st
    fileName: pumps.st

meta:
  author: Alex
  importedFrom: ""
  importedSheet: ""

11.4. Обязательные разделы MetaGen-документа

- kind
- version
- component
- params
- data
- code
- generation

11.5. Назначение разделов

component:
- идентификатор;
- имя;
- тип;
- принадлежность модулю;
- описание.

params:
- параметрическая таблица, максимально близкая по форме к инженерному Excel.

data:
- таблица данных, сигналов, переменных и их атрибутов.

instances:
- список инстансов, если используется в шаблоне.

code:
- исходный шаблон генерации.

generation:
- настройки генерации и указание внешнего движка.

meta:
- дополнительные служебные данные.

11.6. Ключевое правило для MetaGen

В code.text хранится исходный шаблон генерации, а не результат генерации ST.

Сгенерированный ST должен сохраняться отдельно, например:
- generated/pumps.st

Это правило фиксируется жёстко, чтобы не смешивать:
- шаблон;
- сгенерированный результат;
- возможные ручные правки результата.

======================================================================
12. ВНЕШНИЙ PYTHON-ДВИЖОК METAGEN
======================================================================

12.1. Принцип интеграции

Python-движок MetaGen остаётся внешним движком,
а не внутренней частью UI-кода MetaPlatform.

MetaPlatform должна:
- подготавливать документ MetaGen;
- вызывать внешний Python-скрипт;
- передавать ему путь к документу или сериализованные данные;
- получать результат генерации;
- сохранять выходной ST-файл;
- отображать результат и/или логи в UI.

12.2. Почему генератор остаётся внешним

Это архитектурное решение принято потому что:
- существующая генераторная логика уже реализована на Python;
- Python-подход соответствует текущей зрелой инженерной практике;
- UI не должен смешиваться с движком генерации;
- Python-движок можно развивать отдельно;
- Python-генератор можно запускать и вне UI;
- позже его можно заменить или масштабировать независимо.

12.3. Bridge для генератора

Для вызова Python-движка должен существовать отдельный мост:
metagenGeneratorBridge.js

Он отвечает за:
- формирование параметров запуска;
- вызов скрипта;
- передачу документа или файла;
- получение stdout/stderr;
- интерпретацию кода возврата;
- выдачу результата в платформу и логгер.

======================================================================
13. METALAB И METAVIEW
======================================================================

13.1. MetaLab

MetaLab должен быть архитектурно сразу предусмотрен как независимый модуль.

На этапе v1 требуется:
- регистрация модуля;
- резервирование типа документов;
- подготовка структуры каталогов;
- возможность последующего подключения существующего Python-движка
  виртуальной наладки;
- место под собственные команды, редакторы и bridge-слой.

13.2. MetaView

MetaView на этапе v1 может быть резервным модулем без полной реализации функционала,
но архитектурно должен быть предусмотрен полностью:
- модуль регистрируется;
- существуют каталоги проекта;
- определяются заделы под документы экранов, трендов, алармов;
- должны существовать точки расширения для editor factory и command handlers.

======================================================================
14. WORKBENCH UI
======================================================================

14.1. Назначение

Workbench — единая рабочая оболочка MetaPlatform.

Он должен содержать:
- верхнюю панель;
- левое дерево проекта;
- панель вкладок;
- область редакторов;
- нижнюю панель логов.

14.2. Базовая структура Workbench

Top Panel
Project Tree | Tabs + Editor Host
Logs Panel

или, в терминах компонентов:

Workbench
|-- Top Panel
|-- Main Area
|   |-- Project Tree
|   `-- Editor Area
|       |-- Tabs
|       `-- Editor Host
`-- Logs

14.3. Назначение зон

Top Panel:
- отображение имени платформы, имени проекта, возможно глобальных действий.

Project Tree:
- навигация по документам проекта.

Tabs:
- список открытых документов.

Editor Host:
- хост активного редактора документа.

Logs:
- отображение платформенных и модульных логов.

======================================================================
15. ДЕРЕВО ПРОЕКТА (PROJECT TREE)
======================================================================

15.1. Назначение

Project Tree должен визуализировать структуру текущего проекта.

15.2. Источник данных

Tree строится по runtime-модели проекта, сформированной ProjectManager
на основе:
- project.yaml
- сканирования каталогов модулей.

15.3. Пример структуры

Комбикормовый завод

15.4. Операции дерева

Tree должен поддерживать:
- открытие документа двойным кликом или кликом;
- активацию уже открытого документа;
- создание документа;
- переименование документа;
- удаление документа;
- контекстные действия по модулю и документу.

======================================================================
16. ВКЛАДКИ (TABS)
======================================================================

16.1. Назначение

Tabs отображают открытые документы.

16.2. Принцип

Каждая вкладка соответствует одному открытому документу
в конкретном редакторе.

16.3. Модель вкладки

Каждая вкладка должна хранить:
- tabId
- moduleId
- documentKind
- documentPath или documentId
- title
- editorRuntime

16.4. Правило открытия

Если документ уже открыт во вкладке, повторное открытие из Tree
не создаёт новую вкладку, а активирует существующую.

16.5. Операции вкладок

Должны поддерживаться:
- открыть;
- активировать;
- закрыть;
- закрыть текущую;
- обновить заголовок после переименования документа.

======================================================================
17. EDITOR HOST И РЕДАКТОРЫ
======================================================================

17.1. Общая идея

Editor Host — это зона Workbench, в которой модуль отображает
редактор открытого документа.

17.2. Кто отвечает за редактор

Платформа отвечает только за хост и жизненный цикл вкладки.
Сам редактор создаётся модулем.

17.3. Пример

Для документа metagen.component платформа находит модуль MetaGen
и вызывает его openDocument(...), а MetaGen создаёт редактор:
- таблица параметров;
- таблица данных;
- поле шаблона кода;
- кнопка или команда генерации.

17.4. Контракт редактора

Редактор должен:
- уметь монтироваться в DOM-контейнер;
- работать с документом;
- уведомлять о сохранении и изменении;
- освобождать ресурсы при закрытии;
- предоставлять runtime/dispose при необходимости.

======================================================================
18. МОДУЛЬНАЯ АРХИТЕКТУРА
======================================================================

18.1. Общая идея

MetaPlatform должна работать с модулями как с самостоятельными единицами.

18.2. Список модулей v1

- MetaGen
- MetaLab
- MetaView

18.3. Принцип подключения

Платформа не должна жёстко знать детали каждого модуля.
Для этого вводится ModuleRegistry.

======================================================================
19. КОНТРАКТ МОДУЛЯ
======================================================================

Каждый модуль должен реализовать единый контракт.

19.1. Минимальный обязательный контракт

Модуль обязан предоставить:
- id
- name
- documentKinds
- createDefaultDocument(...)
- commands
- openDocument(...)

19.2. Рекомендуемый контракт

{
  id: 'metagen',
  name: 'MetaGen',
  documentKinds: ['metagen.component'],

  createDefaultDocument(context) {},
  commands: {
    CREATE_METAGEN_DOCUMENT: async (...) => {},
    GENERATE_METAGEN_CODE: async (...) => {},
    VALIDATE_METAGEN_DOCUMENT: async (...) => {}
  },

  openDocument({ document, mountElement, context }) {}
}

19.3. Смысл элементов

id:
- внутренний идентификатор модуля.

name:
- отображаемое имя модуля.

documentKinds:
- список типов документов, которыми модуль владеет.

createDefaultDocument(...):
- фабрика создания нового документа модуля.

commands:
- набор обработчиков модульных команд.

openDocument(...):
- фабрика или процедура открытия документа в редакторе модуля.

======================================================================
20. РЕЕСТР МОДУЛЕЙ (MODULE REGISTRY)
======================================================================

20.1. Назначение

moduleRegistry.js — центральный каталог модулей платформы.

20.2. Задачи ModuleRegistry

Он должен:
- регистрировать модули;
- выдавать модуль по id;
- выдавать список всех модулей;
- искать модуль по documentKind;
- выдавать обработчик команды;
- в перспективе поддерживать регистрацию editor factories и иных расширений.

20.3. Внутренние индексы

Внутри ModuleRegistry должны поддерживаться:
- modulesById
- documentKindToModuleId
- commandTypeToHandler

20.4. Основные методы

- registerModule(module)
- getModule(moduleId)
- getAllModules()
- findModuleByDocumentKind(kind)
- getCommandHandler(commandType)

20.5. Принцип

Платформа должна работать с модулем через ModuleRegistry,
а не через прямые if/else по именам модулей.

======================================================================
21. СИСТЕМА КОМАНД
======================================================================

21.1. Решение

В MetaPlatform v1 обязательно реализуется единая система команд.

21.2. Назначение

Любое действие в системе должно выполняться как команда.

Это касается:
- проекта;
- документов;
- вкладок;
- модульных действий;
- генерации;
- сценариев;
- будущих hotkeys и меню.

21.3. Почему система команд обязательна

Она нужна для того, чтобы:
- не дублировать логику между UI-элементами;
- не размазывать действия по разным кнопкам и контролам;
- связать вместе Tree, Tabs, меню, hotkeys и модули;
- централизовать логирование действий;
- дать чистую архитектуру расширения платформы.

21.4. Формат команды

{
  type: "COMMAND_NAME",
  payload: {},
  meta: {
    source: "project-tree"
  }
}

Обязательное поле:
- type

Рекомендуемые поля:
- payload
- meta

21.5. Command Bus

В платформе должен существовать центральный сервис:
commandBus.js

21.6. Обязанности Command Bus

- принимать команду;
- находить обработчик;
- выполнять обработчик;
- логировать выполнение;
- возвращать результат;
- обрабатывать ошибки.

21.7. Принцип работы

UI событие
  -> CommandBus.execute(command)
  -> обработчик платформы или модуля
  -> изменение проекта / UI / документа / внешнего движка

31. ЖИЗНЕННЫЙ ЦИКЛ ДОКУМЕНТА

CREATE_METAGEN_DOCUMENT

31.2. Открытие
Открытие всегда идёт через платформенную команду:
OPEN_DOCUMENT

31.3. Редактирование
Редактирование происходит в редакторе модуля.

31.4. Сохранение
Сохранение выполняется в YAML-файл соответствующего модуля.

- отображаемое имя;

Удаление должно удалять документ из проекта
и обновлять дерево и вкладки.

в code.text исходного YAML-документа.

36. ЧТО НЕ ДОЛЖНО БЫТЬ В РАМКАХ ДАННОГО ТЗ
======================================================================

В рамках текущего ТЗ не требуется:
- облачная синхронизация;
- база данных проекта;
- многопользовательская работа;
- marketplace плагинов;
- сложная RBAC-модель;
- CI/CD;
- распределённое выполнение;
- full undo/redo на уровне всей платформы;
- сложный event sourcing;
- web backend / REST API как обязательная часть v1.

Эти вещи могут быть предусмотрены архитектурно,
но не являются обязательной частью реализации v1.

38. ФИНАЛЬНОЕ РЕШЕНИЕ, ЗАФИКСИРОВАННОЕ ДАННЫМ ТЗ
======================================================================

В рамках данного ТЗ окончательно принимаются следующие архитектурные решения:

1. Имя платформы: MetaPlatform.
2. MetaPlatform является контейнером/Workbench, а не модулем MetaGen.
3. В системе реализуется сущность проекта в виде директории с project.yaml.
4. Документы модулей хранятся в подпапках проекта.
5. MetaGen хранит всё в одном YAML-документе:
   - параметры,
   - данные,
   - инстансы,
   - шаблон кода,
   - настройки генерации.
6. Сгенерированный ST хранится отдельно в generated/.
7. Python-генератор остаётся внешним движком и вызывается из платформы.
8. Реализуется Module Registry.
9. Реализуется Command Bus.
10. Все действия платформы и модулей выполняются через команды.
11. MetaGen, MetaLab и MetaView подключаются как независимые модули.
12. Платформа не должна быть жёстко завязана на MetaGen.
13. Реализация сразу строится в полной архитектуре v1,
    без упрощённой временной модели.
