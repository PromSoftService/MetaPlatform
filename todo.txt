1. ОБЩИЕ ПРИНЦИПЫ, КОТОРЫЕ СЧИТАЮТСЯ ЗАФИКСИРОВАННЫМИ

1.1. Архитектурное направление не меняем
- MetaPlatform остается контейнером / Workbench, а не “обвязкой вокруг MetaGen”.
- Сохраняется полная архитектура v1:
  - Core
  - Workbench
  - Modules
  - Project
- Сохраняется модульность:
  - MetaGen
  - MetaLab
  - MetaView
- Сохраняется принцип:
  - платформа не должна быть жестко завязана на MetaGen
  - MetaLab и MetaView пока могут быть каркасами, но архитектурно учитываются сразу.

1.2. Проект остается директорией
- Проект = папка.
- В проекте есть:
  - project.yaml
  - metagen/
  - metalab/
  - metaview/
  - generated/
- generated/ считается папкой результатов генерации, а НЕ частью обычного канонического save-flow проекта.

1.3. project.yaml
- project.yaml обязателен.
- project.yaml НЕ хранит полный список документов проекта.
- Список документов проекта определяется сканированием папок модулей.
- При этом project.yaml НЕ считается полным source of truth для структуры проекта.
- Структура платформы и набор модульных папок фиксированы архитектурой.

1.4. Имя проекта
- Имя проекта = имя файла project.yaml / имя project file.
- Переименование проекта поддерживается только через Save As / сохранение как другой проект.
- Поле имени проекта как отдельная независимая identity-сущность не вводится.

1.5. Identity документа
- Единственная каноническая identity документа в runtime = GUID.
- Вся внутренняя логика приложения должна строиться только на GUID.
- name документа, file name, path и т.п. не являются identity и не должны использоваться как identity.

1.6. Обязательное правило по тестам
- После любых изменений всегда прогоняется ВЕСЬ набор тестов целиком.
- Не допускается частичный прогон “только рядом затронутых тестов” как финальная проверка.
- Это постоянное правило проекта.

2. ПРИНЯТЫЕ ИСПРАВЛЕНИЯ

2.1. Исправить save-семантику по коллизиям имен
Текущее нежелательное поведение:
- система молча разруливает коллизии через автопереименование файлов
- может появляться -1, -2 и т.д.

Принятое решение:
- убрать автоколлизии / автопереименование при обычном save проекта
- save должен работать как каноническая перезапись project-owned содержимого
- при save:
  - очищать metagen/
  - очищать metalab/
  - очищать metaview/
  - затем заново записывать документы из текущего runtime-состояния
- generated/ НЕ очищать при обычном save
- пользовательские файлы вне metagen/, metalab/, metaview/ не трогать
- generated/ остается отдельной зоной результатов генерации

Смысл:
- диск должен быть точной проекцией текущего project runtime по модульным документам
- generated/ не должен страдать от обычного сохранения проекта
- save проекта и generation output должны оставаться разными семантиками

2.2. Убрать размазанную и дублирующуюся path-логику
Проблема:
- path-логика и file-path семантика размазаны и дублируются в нескольких местах
- это создает риск расхождения поведения

Принятое решение:
- вынести все правила работы с путями проекта в единый слой / единый модуль
- сделать один source of truth для project path semantics

В этот слой должно входить:
- нормализация путей
- join path
- построение project file path
- определение корня проекта
- определение project-owned директорий
- определение путей модульных документов
- общая семантика “что принадлежит проекту, а что нет”

Правило:
- UI и ProjectManager не должны вручную собирать и дублировать path-логику
- любые операции с путями должны идти через единый utility/service слой

2.3. Развязать logger от DOM, сохранив функционал и уровни логов
Проблема:
- logger напрямую завязан на DOM
- это делает core-часть менее чистой и хуже тестируемой

Принятое решение:
- разделить логирование на:
  1) core logger
  2) sinks / adapters

Что должно получиться:
- core logger:
  - знает уровни логов
  - создает log entries
  - фильтрует по level
  - диспатчит записи в sinks
- UI sink:
  - рисует логи в панель Logs / DOM
- console sink:
  - при необходимости зеркалит в console
- memory sink:
  - можно использовать в тестах

Обязательные требования:
- полностью сохранить текущий функционал лог-панели
- полностью сохранить уровни логов
- полностью сохранить понятную работу info / warn / error / debug / none
- сохранить возможность очистки логов
- убрать прямую зависимость core logger от document / DOM API

Правило:
- core пишет в logger
- UI только подписывается на logger и отображает записи

2.4. Исправить create_project_structure.py
Проблема:
- скрипт неверно считает количество созданных директорий
- сейчас счетчик увеличивается не по реально созданным директориям, а по более грубой логике

Принятое решение:
- считать только реально созданные директории
- если mkdir(parents=True) создает несколько уровней, учитывать именно реально появившиеся уровни
- не завышать статистику
- корень проекта не считать лишний раз

Ожидаемый результат:
- итоговая статистика скрипта должна отражать реальное количество созданных директорий, а не “уникально встретившиеся parent paths”

3. ПРИНЯТЫЕ АРХИТЕКТУРНЫЕ РЕШЕНИЯ ПО СПОРНЫМ ПУНКТАМ

3.1. Project Tree: добавить настоящий root-node
Принято:
- в дереве должен быть настоящий корневой узел
- этот узел должен называться НЕ именем проекта, а постоянным именем:
  - Проект

Почему:
- если root-node будет дублировать имя проекта, оно будет повторяться и в header, и в дереве
- это избыточно
- root-node нужен как структурный корень, а не как повтор header

3.2. Header панели проекта сохраняется
Принято:
- верхний header над деревом остается
- header выполняет роль status-line

Header должен:
- быть пустым, если проект не открыт
- показывать имя проекта, если проект открыт
- показывать dirty-state / признак несохраненности

То есть:
- header = состояние проекта
- root-node “Проект” = структурный корень дерева

Это два разных уровня UI, не дублирующих друг друга по смыслу.

3.3. Стилизовать дерево “как надо”
Принято:
- root-node Проект нужно не просто добавить, а нормально стилизовать все дерево
- визуально различать типы узлов

Нужно сделать:
- отдельный стиль для root-node “Проект”
- отдельный стиль для модулей
- отдельный стиль для документов MetaGen / компонентов
- отдельный стиль для сценариев MetaLab
- отдельный стиль для экранов MetaView

Нужно проверить и исправить:
- сейчас шрифт у модулей меньше, чем у компонентов — привести к осмысленной иерархии
- выделенный узел сейчас становится белым — это неверно
- selected state должен быть темнее, чем hover
- hover и selected должны быть четко различимы
- иерархия должна читаться сразу без всматривания

Допустимо и рекомендуется:
- добавить иконки типов узлов:
  - Проект
  - модуль
  - компонент / документ MetaGen
  - сценарий
  - экран

Важно:
- кнопки добавления / удаления компонентов оставить как есть
- действия на узлах не ломать
- не превращать эту задачу в редизайн всего приложения
- задача именно в том, чтобы дерево стало правильным, читаемым и структурно понятным

3.4. GUID как единственная identity
Принято окончательно:
- GUID = единственный канонический идентификатор документа
- все runtime-операции должны работать через GUID

Это означает:
- tree node identity строится по GUID
- tab identity строится по GUID
- active/open/update/delete/select lookup должен опираться на GUID
- path change не должен менять identity документа
- rename display name не должен менять identity документа
- file name не должен менять identity документа

Что НЕ является identity:
- document name
- file name
- path
- label в дереве
- имя вкладки

Они все считаются:
- display attributes
- storage attributes
- presentation attributes
но не identity

3.5. Различия document shapes допустимы, но не должны размазываться по общей логике
Принято:
- различия между component / scenario / screen допустимы и нормальны
- но общая логика приложения не должна в разных местах опираться на name/path как на identity

Смысл:
- shape документа может быть разным
- schema payload может быть разной
- но identity-правила едины

Допустимая модель:
- GUID = identity
- module/kind = routing
- name = display
- path/file name = storage

5. ВАЖНЫЕ УТОЧНЕНИЯ, ЧТОБЫ НЕ ПРОТИВОРЕЧИТЬ САМИМ СЕБЕ

5.1. generated/ не очищать при обычном save
Это зафиксировано отдельно и окончательно:
- generated/ = результаты генерации
- save проекта != regenerate outputs
- обычный save проекта не должен очищать generated/

5.2. project.yaml не является полной моделью проекта
Одновременно зафиксировано:
- project.yaml обязателен
- project.yaml не хранит список документов
- project.yaml не задает полностью структуру проекта
- структура проекта определяется архитектурой платформы + сканированием папок

5.3. Имя проекта и identity проекта
Зафиксировано:
- имя проекта для UI и статуса определяется именем project file
- переименование проекта = Save As
- не вводить вторую конкурирующую семантику имени

8. КОРОТКАЯ ФИНАЛЬНАЯ ФОРМУЛИРОВКА

MetaPlatform развивается как фиксированная модульная инженерная платформа с проектом в виде директории.
Identity документов строится только на GUID.
Обычный save проекта канонически переписывает только модульные YAML-папки проекта и не трогает generated/.
Project Tree получает настоящий root-node “Проект”, при этом верхний header сохраняется как отдельная status-line.
Path-логика централизуется.
Logger переводится в модель core + sinks.
Подсчет директорий в create_project_structure.py исправляется.
После любых изменений всегда прогоняются все тесты целиком.
